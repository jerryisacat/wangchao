# 望潮（Wangchao）前端设计规范

> 本文档定义 `apps/web` 的视觉语言、交互规则和页面组合方式。
>
> 产品目标、数据模型和功能边界仍以 `SPEC.md` 为准；技术架构仍以 `REFACTOR_PLAN.md` 和 `CODEGUIDE.md` 为准。本文只约束前端表现层。

## 1. 设计方向

望潮的前端风格采用 **Kinetic Intelligence**：从 Kinetic Typography 中继承高对比、巨型文字、硬边网格、酸黄色强调和强节奏感，但将持续动效和海报化表达收敛到品牌层、空状态和关键反馈场景。

望潮不是新闻门户，也不是营销落地页。它是一个高频使用的主题情报工作台，因此前端必须同时满足两件事：

- 第一眼有记忆点：像情报终端，而不是通用 SaaS 模板。
- 长时间可阅读：列表、详情、反馈和治理界面必须稳定、密集、低干扰。

```text
品牌层：强 typography、酸黄、硬边、巨大数字、少量运动
工作层：密集信息、稳定阅读、明确状态、低动效
反馈层：用视觉变化表达“系统学到了什么”
```

当前首页重构方向：**首页不是系统控制台，而是未读情报阅读流**。用户进入产品后应先看到需要处理的信息，而不是组织权限、用量审计、处理管线、信源治理和表单堆叠。系统能力可以存在，但必须退到顶部入口、二级页面或情报卡片背后的解释层。

## 2. 适用边界

| 场景 | 风格强度 | 规则 |
|---|---:|---|
| 登录前品牌页 / 空状态 / 新建主题首屏 | 高 | 可以使用巨型标题、横向节奏、强对比块面。 |
| 首页未读情报流顶部 | 低 | 只服务阅读入口、主题筛选和关键操作，不做指标墙。 |
| 情报流列表 | 低 | 稳定行高、清楚分组、少动效，禁止持续 marquee。 |
| 情报详情 | 低 | 以阅读为主，强调来源、解释、反馈动作。 |
| 信源治理 | 中 | 允许硬边卡片、质量分大数字、明显状态色。 |
| 偏好记忆 | 中 | 用权重、置信度和变化方向表达学习过程。 |
| 简报导出 | 中 | 可以偏 editorial / briefing 风格，保持可复制和可读。 |
| 任务状态 / 用量审计 | 低 | 放在二级设置或系统页面，首页不直接展示。 |

## 3. 视觉原则

1. **Typography first**
   - 标题、数字和状态标签构成主要视觉结构。
   - 图标只辅助识别，不承担装饰职责。
   - 情报正文和摘要必须保持正常大小写，不做全大写。

2. **Hard surface, soft reading**
   - 外框、分隔、按钮和状态块可以锐利。
   - 长文本、情报摘要和解释文案必须舒适可读。

3. **Motion as signal**
   - 动效只用于状态变化、反馈确认、系统处理中和品牌节奏。
   - 不用持续运动抢占用户注意力。
   - 所有动效必须尊重 `prefers-reduced-motion`。

4. **Density over decoration**
   - 工作台优先展示可行动信息。
   - 不使用装饰性渐变球、玻璃拟态、过度阴影或无意义插图。

5. **Explainability visible**
   - 每条情报都应能看到为什么推荐、来源可信度、评分和反馈入口。
   - 偏好记忆必须让用户知道系统学到了什么。

## 4. Design Tokens

### 4.1 色彩

当前 `apps/web/src/app/globals.css` 已有暗色基础。后续改造应统一成下列语义 token，而不是在组件里散落 hex。

| Token | 建议值 | 用途 |
|---|---|---|
| `--background` | `#09090b` | 全局背景。 |
| `--foreground` | `#fafafa` | 主文字。 |
| `--surface` | `#121216` | 卡片、面板。 |
| `--surface-strong` | `#18181b` | 重点面板、hover 面。 |
| `--muted` | `#27272a` | 次级块面、背景数字。 |
| `--muted-foreground` | `#a1a1aa` | 次级说明文字。 |
| `--border` | `#3f3f46` | 结构边框。 |
| `--accent` | `#dfe104` | 品牌强调、主 CTA、重点状态。 |
| `--accent-foreground` | `#000000` | 强调色上的文字。 |
| `--cyan` | `#0e7490` | 信息类辅助色。 |
| `--success` | `#22c55e` | 成功、已启用、已保存。 |
| `--warning` | `#f97316` | 候选、观察、待处理。 |
| `--danger` | `#ef4444` | 拒绝、错误、忽略。 |

使用规则：

- `--accent` 只用于主要行动、关键数字、焦点环和重要状态，不铺满所有界面。
- 情报正文禁止使用低对比灰色；说明文字可用 `--muted-foreground`。
- `success / warning / danger` 只能表达状态，不作为装饰色。
- 禁止引入多套品牌渐变；望潮主视觉应保持平面、高对比。

### 4.2 字体

| 层级 | 字体 | 规则 |
|---|---|---|
| 默认 UI | Geist 或系统无衬线 | 保持当前 Next.js 轻量路径。 |
| Display 可选增强 | Space Grotesk | 仅在后续明确引入字体资源时使用。 |
| 数字 / 技术状态 | Geist Mono | 用于分数、任务 ID、用量、时间戳。 |

字号建议：

| 用途 | 建议 |
|---|---|
| 品牌/空状态大标题 | `clamp(3rem, 10vw, 9rem)` |
| 首页页面标题 | `clamp(1.75rem, 4vw, 3rem)` |
| 区块标题 | `1rem - 1.25rem`，高密度场景不要过大 |
| 情报标题 | `1rem - 1.25rem` |
| 情报摘要 | `0.875rem - 1rem` |
| 辅助说明 | `0.75rem - 0.875rem` |
| KPI 数字 | `2rem - 5rem`，视容器大小收敛 |

大小写规则：

- 品牌层标题、按钮、标签可以 uppercase。
- 中文标题不强制转换，保持自然可读。
- 英文 acronym 保留原写法，如 RSS、AI、API、COMAC。
- 情报标题、摘要、来源名称、解释文案不得全大写。

### 4.3 形状和边框

| 元素 | 规则 |
|---|---|
| 主工作台面板 | `border: 1px solid var(--border)`，半径 4px 或 8px。 |
| 强风格品牌区 | 可使用 0px 半径和 2px 边框。 |
| 按钮 | 默认 6px-8px 半径；强 CTA 可 0px-4px。 |
| 指标卡 | 稳定尺寸，避免数字变化造成布局跳动。 |
| 情报列表行 | 边框/分隔线优先，不用阴影。 |

原始 Kinetic Typography 的 `0px radius` 不能全局照搬。望潮的阅读界面允许少量半径，避免工具界面过于刺眼。

### 4.4 间距和密度

- 工作台外边距：`20px - 24px`。
- 卡片内边距：普通 `16px`，重点 `20px - 24px`。
- 表单间距：`12px - 16px`，不要使用落地页式超大垂直间距。
- 首页页面网格：顶部导航 + 中间限宽单列阅读区是首选结构。
- 管理页面网格：信源、设置、审计等二级页面可使用左导航、多列或上下文面板。
- 移动端必须原生支持：单列阅读、顶部或抽屉导航、safe-area padding、无横向滚动，所有高频动作以 44px 以上触达目标呈现，不允许把桌面 hover 当成唯一入口。

## 5. 页面组合

### 5.1 首页重构计划：未读情报流

首页的唯一核心任务是让用户处理未读情报。

目标结构：

```text
顶部导航
  左侧：望潮
  中间：未读情报 / 今日简报 / 已保存 / 偏好
  右侧：新增主题 / 信源管理 / 主题 / 设置

中间限宽阅读区
  页面标题：未读情报
  轻量状态：未读数量、当前主题筛选、最近更新时间
  情报流：按重要度、时间和个人偏好排序
  收尾状态：全部处理完或暂无新情报
```

首页保留：

- 未读情报流。
- 主题筛选：全部主题 / 单个主题。
- 情报卡片：标题、摘要、为什么重要、来源、时间、主题。
- 快速动作：已读、收藏、减少这类、原文。
- 顶部入口：新增主题、信源管理。

首页移出：

- 组织与权限。
- 用量审计。
- 处理管线。
- 信源治理详情。
- 偏好记忆详情。
- 新建主题大表单。
- 大面积 KPI 指标卡。

宽度规则：

- 主阅读区建议 `max-width: 760px - 920px`。
- 页面外层允许留白，不用为了填满屏幕而增加侧栏。
- 桌面端可以在情报详情页使用二栏；首页默认不使用右侧详情栏。
- 移动端保持单列，顶部操作可折行但不能挤压标题。

### 5.2 App Shell

首页 App Shell 使用顶部导航，而不是固定左侧控制台。

```text
顶部导航
  - 望潮品牌
  - 未读情报
  - 今日简报
  - 已保存
  - 偏好
  - 新增主题
  - 信源管理
  - 主题
  - 设置
```

规则：

- 顶部菜单必须是实际入口，不做无行为的伪导航。
- `新增主题` 和 `信源管理` 是按钮或链接，不在首页展开复杂管理表单。
- `主题`、`设置` 等二级入口必须保留文字标签和当前页状态，不能只依赖难发现的图标。
- 品牌 mark 可以使用酸黄和硬边，但不应抢情报流注意力。
- 搜索可以作为轻入口存在，但不能让首页变成命令面板。
- 系统状态、任务、组织、审计不进入首页顶层导航。

### 5.3 情报流

情报流是核心页面，目标是快速判断：

- 这条情报讲什么？
- 为什么重要？
- 来自哪里？
- 系统为什么把它排上来？
- 我要读、存、忽略还是纠偏？

首页情报卡片应包含：

| 信息 | 要求 |
|---|---|
| 主题 | 显示主题名称或短标签，帮助用户判断上下文。 |
| 来源和时间 | 显示 source name、原文链接、发布时间或抓取时间。 |
| 标题 | 两行以内，超出进入详情。 |
| 摘要 | 2-4 行，说明发生了什么。 |
| 为什么重要 | 用用户语言解释影响、对象和后续跟踪价值。 |
| 排序解释 | 轻量展示，不使用工程化 `matched keywords` 原文。 |
| 动作 | 已读、收藏、减少这类、原文。 |

弱化或隐藏：

- `gravityScore`、`score` 等工程分数不在卡片里抢主视觉，可作为详情里的解释信息。
- `read / saved / dismissed` 等状态可以转成中文动作和状态，不直接暴露枚举名。
- `Item`、`TaskRun`、`UsageEvent`、`PreferenceMemory` 等内部对象不得出现在首页文案中。

交互规则：

- hover 可以轻微提升边框或背景，不做整卡酸黄反色。
- saved 状态可以使用酸黄或成功色点亮。
- dismissed 使用低对比或移出默认列表。
- 支持键盘焦点，焦点样式必须明显。
- 已读动作完成后，卡片可以从未读流移出或进入轻量撤销状态。

### 5.4 情报详情

详情面板采用稳定阅读布局：

```text
标题
摘要
为什么重要
影响对象
来源与原文链接
系统解释
反馈动作
导出入口
```

反馈动作必须区分“忽略此条”和类别偏好：“忽略此条”改变事件状态并退出默认信息流；“多关注这类 / 少关注这类”只调整当前 Topic 下的 category 偏好，提交后留在详情页并显示明确反馈。移动端允许动作区换行，每个按钮仍保持至少 44px 触达高度。

摘要区域必须读取结构化状态：采集等待、采集失败、内容不足、平台暂不支持和 AI 失败均显示明确提示，不得用标题或 RSS 摘要伪装成 AI 摘要；这些状态仍保留“原文”和“重新采集”动作。首页情报卡片使用同一状态文案，避免列表与详情含义不一致。

详情页可以用更强的标题层级，但正文必须保持阅读宽度。不要使用滚动视差、持续 marquee 或大面积 hover 反色。

### 5.5 新建主题

新建主题是最适合强化 Kinetic 风格的页面。

推荐结构：

```text
巨型标题：TRACK WHAT MATTERS
自然语言输入：我想关注……
AI 生成主题草案
include / exclude / entities / keywords
确认创建
```

设计规则：

- 输入框可以更大、更有张力。
- 草案编辑区必须回到清晰表单，不做海报效果。
- 创建后展示“系统开始观察”的反馈动效，但需要 reduced-motion 版本。
- 首页只保留 `新增主题` 入口；不要在首页直接展开完整创建表单。

### 5.6 信源管理

信源管理可以比情报流更硬朗，因为它偏操作面板。首页只显示情报来源，不展示信源治理指标。

每个信源卡片显示：

- 状态：candidate / active / muted / rejected。
- 命中率、噪音率、重复率、质量分。
- 最近抓取时间。
- 推荐动作：approve / observe / mute / reject。
- 证据：为什么推荐这个操作。

这里可以使用大数字和强边框，但 hover 反色只应用于可点击动作，不应用于整段文本。

### 5.7 偏好记忆

偏好记忆必须可解释，不做神秘 AI 黑盒。

展示规则：

- 权重增加：使用正向颜色和上升标记。
- 权重降低：使用警示/低对比标记。
- 置信度：用进度条或数字，不夸张。
- 解释：明确来自哪些反馈信号。

示例信息：

```text
category:keyword:AI
weight +2.5
confidence 0.71
3 feedback signals increased this preference.
```

### 5.8 简报和导出

简报页面应更接近 editorial / intelligence report：

- 标题清楚。
- 生成时间明确。
- 来源链接保留。
- Markdown 导出入口显著。
- 不把简报正文做成全大写或巨型 typography。
- **周期筛选**：简报中心页面顶部提供 DAILY/WEEKLY/MONTHLY 周期筛选 tabs（≥44px touch target），使用 `data-active` 属性标识当前选中项。
- **Obsidian-friendly 文件名**：简报导出文件名格式为 `{date}-{period}-{slug}.md`（如 `2026-07-11-weekly-ai-infrastructure.md`），便于 Obsidian/文件系统排序。
- **主题批量导出**：主题详情页提供"批量导出"按钮，下载主题 Top 100 事件为单个 Markdown 文件（`{date}-batch-{slug}.md`）。
- **主题时间线**：主题详情页提供"时间线"入口，时间线页面按 `occurredAt` 倒序展示主题全部正式事件，包含 merged sources 信息。

### 5.9 管理后台设置

#### AI 凭证表单增强

- **模型嗅探**：AI 凭证表单中，点击"刷新模型列表"按钮会调用 `/admin/settings` 的 `listAiModelsAction`，嗅探 OpenAI-compatible 端点的可用模型列表。成功后以 `<select>` 下拉框展示模型 ID 及 `ownedBy` 信息，支持"自定义..."选项回退到自由输入 `<Input>`。嗅探结果不持久化，刷新页面后需重新获取。
- **自定义 Provider 手动确认**：当选择"自定义" provider 且自动测试失败时，显示"我已确认此 Key 有效" checkbox。勾选后跳过自动测试门控，允许直接保存。取消勾选会清空测试状态，需要重新测试。
- **计费提示**：AI 凭证测试按钮下方显示"测试将发送一次最小 API 请求，可能产生极少量费用"提示文案。
- **独立性说明**：页面顶部显示"AI 凭证与搜索凭证相互独立，可分别保存与清除。配置其中一项不会影响另一项。"
- **Telegram 即时推送**：Telegram tab 分开展示凭证和即时推送开关；Free 显示升级提示，未配置凭证不可开启，服务端重复执行权限与计划校验。

## 6. 组件规范

### 6.1 Button

| Variant | 用途 | 风格 |
|---|---|---|
| `primary` | 新主题、确认、导出 | 酸黄底，黑字，强焦点。 |
| `secondary` | 普通动作 | 深色面，边框。 |
| `ghost` | 图标、轻操作 | 无背景，hover 轻微变化。 |
| `danger` | 忽略、拒绝、删除类 | 危险色只在必要时出现。 |

按钮要求：

- 最小点击区域 44px。
- 有 `hover`、`focus-visible`、`active`、`disabled` 状态。
- 图标按钮必须有 `aria-label`。
- 按钮文字不允许在窄宽度内挤成不可读的多行。

### 6.2 Card

卡片分两类：

- `work-card`：情报、详情、治理、设置，强调可读性。
- `kinetic-card`：空状态、品牌区、关键数字，强调视觉记忆。

`work-card` 不做整卡大反色；`kinetic-card` 可以使用更硬的边框和更强的 hover。

### 6.3 Badge

Badge 用于状态，不用于装饰堆叠。

| 状态 | 语义 |
|---|---|
| success | active、saved、done |
| warning | candidate、observe、pending |
| danger | rejected、error、dismissed |
| muted | read、archived、skipped |
| accent | selected、primary signal |

### 6.4 Form

表单用于创建主题、绑定 RSS、候选信源。

规则：

- label 永远显示，不只依赖 placeholder。
- 错误信息靠近字段。
- URL 输入必须清楚标注。
- 创建主题的自然语言输入可以更大；普通设置表单保持密集。

## 7. 动效规则

允许：

- 按钮轻微 scale / press。
- 状态切换时的颜色变化。
- 新建主题成功后的短暂反馈。
- 空状态或品牌页的低速 typography 节奏。
- 数据更新时的轻微 highlight。

谨慎：

- marquee 只用于品牌层或空状态，不能出现在情报流主阅读区。
- scroll scale / parallax 只用于非工作流页面。
- 大面积 hover 反色只用于可操作摘要，不用于正文阅读。

禁止：

- 情报列表持续运动。
- 阅读正文跟随滚动缩放。
- 多个区域同时 marquee。
- 800ms 以上的慢动画。
- 动效作为唯一状态提示。

必须实现：

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    scroll-behavior: auto !important;
    transition-duration: 0.01ms !important;
  }
}
```

## 8. 响应式规则

必须验证宽度：

- 320px
- 375px
- 414px
- 768px
- 1024px
- 1440px

移动端规则：

- 主布局单列。
- 左侧导航不可固定占宽，应变为顶部导航或菜单；横向导航必须可触摸滚动且隐藏滚动条不影响可达性。
- 情报卡片先显示标题、摘要、来源、动作；动作在移动端必须显示文字标签或同等可理解提示，不只给图标。
- 右侧详情面板下移为详情区或独立页面。
- 所有按钮可触达，最小 44px，并考虑 iOS/Android safe area。
- 不能出现横向滚动。
- 巨型标题必须使用 `clamp()`，并允许换行。

## 9. 可访问性

- 所有交互元素必须可键盘访问。
- `focus-visible` 不得被移除。
- 图标按钮必须有 `aria-label`。
- 状态不能只靠颜色表达，必须有文字或图标辅助。
- 情报来源链接使用真实 `<a>`，外链标注清楚。
- 表单字段必须有 label。
- 动效必须尊重 `prefers-reduced-motion`。
- 主内容区应有明确 landmark：`main`、`nav`、`aside`、`section`。

## 11. 不做什么

- 不把整个应用做成营销页。
- 不在情报阅读区使用持续 marquee。
- 不让所有标题和正文 uppercase。
- 不为了风格牺牲信息密度。
- 不引入复杂动画库，除非页面确实需要且验证性能。
- 不把候选源、任务、用量这类工具信息做成装饰性海报。
